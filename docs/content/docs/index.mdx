You send requests to RiftRelay instead of directly to the Riot Games API. It validates the route, maps it to the right rate-limit bucket, queues it if needed, and forwards it when a slot opens up. Your app never sees a `429`.

## What it does

- Proxies requests through `/{region}/{riot-api-path}` — see [usage](/docs/guides/usage)
- Tracks app and method rate limits from Riot response headers
- Spreads requests evenly across the remaining window instead of bursting at the reset boundary (you still use 100% of your quota)
- Supports `X-Priority: high` for latency-sensitive traffic — see [priority requests](/docs/guides/priority-requests)
- Exposes `/healthz`, optional `/metrics`, `/debug/pprof/`, and `/swagger/` — see [endpoints](/docs/reference/endpoints)

## How requests flow

1. Your app sends a request to RiftRelay.
2. The path is parsed as `/{region}/{riot-api-path}`.
3. The upstream path is matched to a Riot API pattern to derive a bucket key.
4. The admission controller decides when the request may go through.
5. The request is forwarded to `{region}.api.riotgames.com`.
6. Rate-limit state is updated from the Riot response headers.

## Configuration

Only `RIOT_TOKEN` is required. Everything else has defaults. See the [configuration reference](/docs/reference/configuration).
