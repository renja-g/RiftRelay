title: Metrics
description: Prometheus-compatible metrics exposed by RiftRelay for queueing, admission control, and upstream request visibility.

---

Enable with `ENABLE_METRICS=true` (on by default). Scrape `GET /metrics`.

```yaml
scrape_configs:
  - job_name: riftrelay
    static_configs:
      - targets:
          - localhost:8985
```

The Docker Compose stack can start Prometheus (`9090`) and Grafana (`3000`) alongside RiftRelay.

## Available metrics

### `riftrelay_http_requests_total` (counter)

Total inbound requests. Labels: `region`, `endpoint`, `priority`.

### `riftrelay_http_inflight` (gauge)

Requests currently being processed. Labels: `region`, `endpoint`, `priority`.

### `riftrelay_admission_total` (counter)

Admission outcomes. Labels: `outcome`, `region`, `endpoint`, `priority`.

Outcome values include `allowed`, `rejected_queue_full`, `rejected_timeout`, `shutting_down`, and generic `rejected`.

### `riftrelay_queue_depth` (gauge)

Current queue depth. Labels: `bucket`, `priority`. If this keeps climbing and doesn't recover, traffic is outpacing your rate-limit budget.

### `riftrelay_queue_wait_seconds` (histogram)

Time requests spend waiting in the admission queue. This is usually the most useful metric for spotting rate-limit pressure — if wait times are climbing, you're hitting capacity.

### `riftrelay_upstream_responses_total` (counter)

Upstream responses from Riot. Labels: `code`, `region`, `endpoint`, `priority`. Watch `code="429"` specifically.

### `riftrelay_upstream_duration_seconds` (histogram)

How long upstream calls take after admission. Separates queue wait (before the call) from upstream latency (during the call), so you can tell whether slowness is from rate limiting or from Riot.

### `riftrelay_request_duration_seconds` (histogram)

End-to-end request duration (queue wait + upstream + overhead). Labels: `region`, `endpoint`, `priority`.

### Go runtime and process

Standard `go_*` and `process_*` collectors are registered.

## Labels

| Label | Values | Used by |
| --- | --- | --- |
| `region` | `europe`, `americas`, `asia`, `na1`, `euw1`, etc. | requests, admission, upstream |
| `endpoint` | normalized Riot API route pattern | requests, admission, upstream |
| `bucket` | limiter bucket key | queue depth, queue wait |
| `priority` | `normal`, `high` | all |
| `outcome` | `allowed`, `rejected_*`, `shutting_down` | admission |
| `code` | HTTP status code | upstream responses |

## Useful PromQL

Upstream 429 rate (are you hitting Riot's limits?):

```text
rate(riftrelay_upstream_responses_total{code="429"}[5m])
```

Admission rejection rate:

```text
sum(rate(riftrelay_admission_total{outcome!="allowed"}[5m]))
```

p99 queue wait time:

```text
histogram_quantile(0.99, sum(rate(riftrelay_queue_wait_seconds_bucket[5m])) by (le))
```

p99 upstream latency:

```text
histogram_quantile(0.99, sum(rate(riftrelay_upstream_duration_seconds_bucket[5m])) by (le))
```

Queue depth by bucket (spot hotspots):

```text
topk(10, riftrelay_queue_depth)
```

## Reading the numbers

**Healthy:** queue depth near zero, low wait times, rare rejections, upstream `200`s dominate, bursts settle quickly.

**Rate-limit pressure:** queue depth growing, wait times climbing, more rejections and `429`s, slow recovery after spikes.

**Upstream problems (not rate limiting):** queue wait is fine but upstream duration or error codes are elevated. That's Riot API latency or an outage, not your limiter.

## Troubleshooting

**`/metrics` returns 404** — `ENABLE_METRICS` is probably `false`.

**Prometheus can't scrape** — check that RiftRelay is running, port is right, and firewall allows it.

**Grafana shows no data** — verify the Prometheus data source is configured and scraping successfully.

**Queue depth stays high** — traffic is arriving faster than it can be admitted. Look at which buckets are hot and whether `X-Priority: high` is overused.
