RiftRelay exposes Prometheus-compatible metrics at `GET /metrics`. Enable with `ENABLE_METRICS=true` (see [configuration](/docs/reference/configuration)).

## What the metrics give you

Operational visibility into how RiftRelay behaves under load:

- request queue depth
- queue wait time
- admission outcomes (allowed, rejected, reason)
- upstream response behavior (status, region, bucket, priority)
- upstream request duration
- standard Go runtime and process metrics

## Prometheus scraping

```yaml
scrape_configs:
  - job_name: riftrelay
    static_configs:
      - targets:
          - localhost:8985
```

The Docker Compose stack can start Prometheus and Grafana alongside RiftRelay. Default ports: RiftRelay `8985`, Prometheus `9090`, Grafana `3000`.

## Metric categories

### Queue depth

How many requests are waiting in a bucket. Use to identify which Riot API buckets are under pressure and whether traffic spikes cause sustained backlog. If queue depth keeps growing and does not recover, traffic exceeds available capacity.

### Queue wait time

How long requests spend in admission control before being allowed or rejected. One of the most important signals. Increasing wait times often indicate rising load, exhausted rate-limit windows, or sustained bursts on a small set of buckets.

### Admission outcomes

Whether requests are allowed or rejected (queue full, timeout, shutdown, etc.). Helps distinguish a healthy queueing system from one that is actively shedding load.

### Upstream request counts

Responses from Riot, grouped by status code, region, bucket, and priority. Answers: are upstream calls succeeding, are `429`s increasing, are certain buckets failing more than others?

### Upstream duration

How long upstream Riot API requests take after admission. Separates **queue wait time** (before the request is sent) from **upstream duration** (once in flight), making it easier to tell whether latency is from rate-limit pressure or Riot API response time.

### Go runtime and process

Standard Prometheus Go and process metrics for service health and capacity observation.

## Labels and dimensions

Depending on the metric, labels may include:

- `region`: Riot routing region (e.g. `europe`, `americas`, `asia`, `na1`, `euw1`)
- `bucket`: normalized limiter bucket (based on Riot API route pattern)
- `priority`: `normal` or `high`
- `status`: HTTP status code for upstream metrics
- `reason`: why a request was allowed or rejected

## How to interpret

**Healthy system:** queue depth near zero, low wait time, rare rejections, upstream success dominates, bursts settle quickly.

**Rate-limit pressure:** growing queue depth, increased wait duration, more rejections and `429`s, prolonged recovery after spikes.

**Exhausted rate-limit capacity:** queues elevated for long periods, wait times increasing during normal traffic.

**Upstream problems (not limiter):** queue wait low but upstream duration or error statuses rise. That points to Riot API latency, outages, or network issues.

## Suggested dashboards

- queue depth by bucket
- queue wait percentiles
- admission result counts by reason
- upstream response counts by status
- upstream duration percentiles
- Go runtime health panels

## Recommended alerts

- sustained queue depth above baseline
- admission rejection rate above threshold
- elevated upstream `429` or `5xx` rate
- unusually high queue wait time
- missing scrapes from `/metrics`

## Security

Expose `/metrics` only to Prometheus or internal monitoring networks. Avoid public edge exposure.

## Troubleshooting

**`/metrics` not found.** Ensure `ENABLE_METRICS=true`.

**Prometheus cannot scrape.** Check RiftRelay is running, host/port is correct, metrics are enabled, firewall allows access.

**Grafana shows no data.** Prometheus may not be scraping, data source may be misconfigured, or metrics disabled.

**Queue metrics high.** Traffic arriving faster than it can be admitted. Investigate bucket hotspots and high-priority overuse.
